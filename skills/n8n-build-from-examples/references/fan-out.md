# Fan-out: раскидывание результата во все источники СРАЗУ

Главное требование шаблона: при валидных данных результат формы уходит **одновременно**
в amoCRM, Telegram и Postgres. В n8n это делается так: один выход ноды подключается к
нескольким целевым нодам — они выполняются параллельно.

## Топология

```
Webhook → Code "Валидация" → IF (valid?)
                               ├ true  →  amoCRM (HTTP)
                               │          Telegram (отправка)
                               │          Postgres (insert)
                               └ false →  Telegram "Алерт"
```

## Как устроены `connections` (ключ — ИМЯ ноды, не id)

В n8n `connections` индексируются по имени исходной ноды. У ноды IF два выхода:
индекс `0` = true, индекс `1` = false. Чтобы получить fan-out, в выходе `true`
кладём **массив из трёх целей**.

```json
{
  "connections": {
    "Webhook": {
      "main": [[{ "node": "Валидация", "type": "main", "index": 0 }]]
    },
    "Валидация": {
      "main": [[{ "node": "IF Валидно", "type": "main", "index": 0 }]]
    },
    "IF Валидно": {
      "main": [
        [
          { "node": "amoCRM", "type": "main", "index": 0 },
          { "node": "Telegram", "type": "main", "index": 0 },
          { "node": "Postgres", "type": "main", "index": 0 }
        ],
        [
          { "node": "Telegram Алерт", "type": "main", "index": 0 }
        ]
      ]
    }
  }
}
```

Разбор:
- `"IF Валидно".main[0]` — выход **true**: массив из ТРЁХ целей → параллельный fan-out.
- `"IF Валидно".main[1]` — выход **false**: одна цель, нода алерта.

## Через MCP-диффы

Если строишь/правишь инкрементально, добавляй связи операциями `addConnection`
в `n8n-mcp:n8n_update_partial_workflow`. Для ветки false указывай соответствующий выход
ноды IF (branch/`sourceIndex`). Точную форму операции сверь через
`n8n-mcp:tools_documentation`.

## Замечания

- Все три ноды-назначения получают на вход один и тот же item с `data` из Code-ноды
  (`$json.data.email` и т.п.) — маппинг полей делаешь в каждой ноде-назначении.
- Если важно, чтобы запись в Postgres шла строго после amoCRM (а не параллельно) — это
  уже не fan-out, а цепочка; по умолчанию требование менеджера = всё сразу.
