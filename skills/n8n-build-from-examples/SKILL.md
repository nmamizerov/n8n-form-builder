# n8n-build-from-examples — сборка воркфлоу через MCP по примерам

Этот скилл вызывается оркестратором `create-form-handler` ПОСЛЕ того, как менеджер
апрувил план. Здесь — техническая часть: как собрать корректный воркфлоу.

Подход — **вариант B**: не подставляй слепо готовый JSON-шаблон. Собирай воркфлоу под
конкретный запрос через MCP, а реальные примеры из `examples/` используй как образец
структуры и стиля (few-shot), чтобы попасть максимально точно.

## Алгоритм

### 1. Изучить примеры
Прочитай `examples/README.md` и выбери 1–2 наиболее близких примера к задаче менеджера.
Прочитай их `.json` (через Read). Это эталон топологии и формата `nodes`/`connections`.
Чем больше релевантных примеров — тем точнее результат.

### 2. Уточнить схемы нод через MCP (страховка от дрейфа версий)
Для каждой ноды, в параметрах которой не уверен, вызови `n8n-mcp:get_node`
(mode `full` или `property-search`). Никогда не доверяй дефолтам — задавай все значимые
параметры явно (главная причина рантайм-ошибок в n8n). Шпаргалки по нодам — в
`references/` этого скилла:
- `references/webhook-node.md` — приём POST с формы
- `references/code-validation.md` — генерация JS-кода валидации по правилам менеджера
- `references/fan-out.md` — как раскидать результат во все источники СРАЗУ (главное требование)
- `references/amocrm-api.md` — amoCRM через community-ноду `n8n-nodes-amocrm` (yatolstoy): связка Contact → Lead
- `references/telegram-node.md` — Telegram как назначение и как канал алертов
- `references/postgres-node.md` — insert в Postgres

### 3. Собрать структуру воркфлоу
Топология обработчика форм (см. `references/fan-out.md` для точной формы `connections`):

```
Webhook(POST /path)
  → Code "Валидация" (возвращает {valid, data, errors})
  → IF (valid === true)
       true  → [amoCRM Create Contact, Telegram, Postgres]   ← все три параллельно
                amoCRM Create Contact → amoCRM Create Lead    ← цепочка (см. amocrm-api.md)
       false → Telegram "Алерт" (текст из errors)
```

amoCRM реализуется ДВУМЯ нодами `n8n-nodes-amocrm.amocrm`: сначала создаём контакт,
затем лид со ссылкой на его id (нода не умеет создать лид с новым контактом в одной операции).

Code-нода генерируется под конкретные поля и правила менеджера (см. `references/code-validation.md`).
В `connections` цели задаются по ИМЕНИ ноды; чтобы получить fan-out, выход `true` ноды IF
содержит массив из трёх целей.

### 4. Валидация ДО деплоя
- `n8n-mcp:validate_node` (mode `minimal`, затем `full`) для каждой ноды.
- `n8n-mcp:validate_workflow` для всего воркфлоу (связи, выражения).
Исправь все ошибки до создания.

### 5. Создание и автофикс
- `n8n-mcp:n8n_create_workflow` — создать воркфлоу.
- `n8n-mcp:n8n_validate_workflow` (по id) — провалидировать уже на инстансе.
- При ошибках — `n8n-mcp:n8n_autofix_workflow`, затем перепроверить.

### 6. Правки — диффами
Если потом нужно что-то изменить — используй `n8n-mcp:n8n_update_partial_workflow`
(батч типизированных операций), а не полную замену.

### 7. Вернуть результат
Сообщи оркестратору: id воркфлоу, путь webhook и итоговый URL вида
`<N8N_API_URL>/webhook/<path>` (и `/webhook-test/<path>` для теста до активации).

## Креды назначений
amoCRM / Telegram / Postgres подключаются через **n8n credentials**, которые менеджер
создаёт в UI n8n один раз. В воркфлоу ноды только ссылаются на эти credentials по имени —
не вписывай секреты в параметры нод. Если нужной credential ещё нет, объясни менеджеру
(через оркестратор), как её создать в n8n.

## Правила
- Все значимые параметры нод — явно. Дефолтам не доверять.
- Валидация обязательна перед `n8n_create_workflow`.
- Секреты — только через n8n credentials, не в JSON воркфлоу.
