# n8n-setup — подключение к n8n через MCP

Цель: гарантировать, что MCP-сервер `n8n-mcp` зарегистрирован и подключён к инстансу n8n,
прежде чем строить воркфлоу. Менеджер нетехнический — объясняй каждый шаг простыми словами
и не показывай ему сырые ошибки без перевода.

## Чеклист (выполняй по порядку)

### 1. Проверить, зарегистрирован ли MCP-сервер

```bash
claude mcp get n8n-mcp
```

- Если сервер найден → переходи к шагу 3 (проверка связи).
- Если команда говорит, что сервера нет → шаг 2.

### 2. Зарегистрировать MCP-сервер

В репозитории плагина уже есть `.mcp.json` с правильным конфигом (`npx n8n-mcp`,
`MCP_MODE=stdio`). Если он подхватился — сервер появится автоматически. Если нет,
зарегистрируй явно:

```bash
claude mcp add --transport stdio n8n-mcp \
  -e MCP_MODE=stdio \
  -e DISABLE_CONSOLE_OUTPUT=true \
  -e LOG_LEVEL=error \
  -e N8N_API_URL="<URL>" \
  -e N8N_API_KEY="<KEY>" \
  -- npx -y n8n-mcp
```

`MCP_MODE=stdio` обязателен — без него дебаг-логи ломают MCP-протокол и появляются ошибки
вида `Unexpected token ...`.

### 3. Получить креды у менеджера (если их ещё нет)

Спроси у менеджера два значения и объясни, где их взять:

- **N8N_API_URL** — адрес его n8n, например `https://acme.app.n8n.cloud`.
- **N8N_API_KEY** — ключ API. Инструкция словами: «Открой n8n → внизу слева Settings →
  раздел **n8n API** → **Create API key** → скопируй и пришли сюда».

Куда записать креды (выбери по ситуации, не клади ключ в git):

- **Предпочтительно** — переменные окружения `N8N_API_URL` / `N8N_API_KEY` (их подставляет
  committed `.mcp.json` через `${...}`). Подскажи менеджеру добавить их в свой shell-профиль,
  либо запиши их в локальный `.env`/`.mcp.local.json` (оба в `.gitignore`).
- **Запасной вариант для совсем нетехнических** — впиши ключ прямо в `.mcp.json` (замени
  `${N8N_API_KEY}` на значение) и обязательно раскомментируй строку `.mcp.json` в `.gitignore`,
  чтобы файл с секретом не попал в репозиторий. Предупреди менеджера об этом.

После изменения конфига MCP-сервер нужно переподключить — попроси менеджера выполнить
`/mcp` или перезапустить сессию, если инструменты `n8n-mcp` ещё не видны.

### 4. Проверить связь

Вызови MCP-инструмент `n8n-mcp:n8n_health_check`.

- OK → сообщи менеджеру «Подключение к n8n работает ✅» и верни управление вызвавшему скиллу.
- Ошибка авторизации (401/403) → ключ неверный или истёк, попроси новый.
- Ошибка соединения → проверь URL (для self-hosted локального n8n нужен
  `N8N_API_URL=http://host.docker.internal:5678` и `WEBHOOK_SECURITY_MODE=moderate`, но у нас
  целевой сценарий — общий n8n Cloud, так что чаще всего дело в опечатке в URL).

## Правила

- Никогда не печатай API-ключ в ответе и не коммить его.
- Не продолжай к созданию воркфлоу, пока `n8n_health_check` не вернул OK.
- Если что-то не получается — объясни менеджеру проблему человеческим языком и предложи
  конкретный следующий шаг, а не стектрейс.
