# create-form-handler — мастер создания обработчика форм в n8n

Ты ведёшь **нетехнического менеджера** за руку. Говори простым языком, без жаргона n8n.
Один шаг — один-два вопроса, не вываливай всё сразу. Воркфлоу создаётся ТОЛЬКО после
явного апрува плана.

Создаваемый воркфлоу всегда: `Webhook → валидация (Code) → если данные верны, отправка
СРАЗУ во все выбранные источники (amoCRM/Telegram/Postgres); если нет — алерт в Telegram`.

## Шаги (выполняй по порядку, отмечай через TodoWrite)

### Шаг 1. Подключение к n8n
Запусти скилл **n8n-setup** (Skill tool). Он проверит/установит MCP-сервер n8n и креды,
убедится что `n8n_health_check` = OK. Не продолжай, пока связь не подтверждена.

### Шаг 2. Выбор шаблона
Сейчас доступен один шаблон — **«Обработчик форм»**. Скажи это менеджеру и подтверди,
что он хочет именно его. (Шаг оставлен явным, чтобы потом добавлять шаблоны.)

### Шаг 3. Собрать требования (вопросы по одному-два)
Спроси и зафиксируй:

1. **Название формы / для чего она** — из него сделаешь slug пути вебхука (латиница,
   напр. «Заявки с лендинга Acme» → `lead-acme`).
2. **Какие поля приходят с формы?** Для каждого: имя поля (как в форме), обязательное или нет,
   тип/правило (email, телефон, число, текст, один из вариантов…). См. соответствие правил
   коду в скилле n8n-build-from-examples → `references/code-validation.md`.
3. **Куда слать результат?** По умолчанию во все три, но подтверди и собери параметры:
   - **amoCRM**: поддомен `<x>.amocrm.ru`, какие поля лида/контакта заполнять. Нужна
     OAuth2-credential amoCRM в n8n.
   - **Telegram**: chat_id рабочего чата, формат сообщения. Нужна Telegram-credential (токен бота).
   - **Postgres**: схема.таблица и колонки. Нужна Postgres-credential.
4. **Куда слать алерты о невалидных заявках?** По умолчанию — тот же Telegram-чат
   (можно отдельный chat_id).

Если у менеджера ещё нет нужных **credentials** в n8n — объясни простыми словами, как их
создать в n8n (Settings/Credentials), и какие данные понадобятся. Секреты он вводит в n8n,
не сюда.

### Шаг 4. Показать ПЛАН и получить апрув
Покажи менеджеру понятный текстовый план (без JSON-внутренностей):

- Название воркфлоу и URL вебхука: `<N8N_API_URL>/webhook/<slug>` (метод POST).
- Список полей и правил валидации простыми словами.
- Куда уходят валидные заявки (перечисли 3 источника и что именно пишется).
- Что происходит при невалидной заявке (алерт в Telegram с описанием ошибок).

Затем спроси явный апрув (через AskUserQuestion: «Создать этот воркфлоу в n8n?» —
варианты: Создать / Изменить / Отмена). **Без выбора «Создать» воркфлоу не создаём.**

### Шаг 5. Создать воркфлоу
После апрува запусти скилл **n8n-build-from-examples** (Skill tool), передав ему собранные
требования. Он соберёт воркфлоу через MCP по примерам из `examples/`, провалидирует и создаст.

### Шаг 6. Отдать результат менеджеру
Сообщи простым языком:
- Воркфлоу создан (название, id).
- **Боевой URL вебхука** для вставки в форму (метод POST, тело — JSON с полями формы).
- **Тестовый URL** (`/webhook-test/<slug>`) для проверки до активации.
- Напомни **активировать** воркфлоу в n8n (тумблер Active), иначе боевой вебхук не работает.
- Краткий совет, как протестировать (отправить тестовую заявку).

## Правила
- Простой язык, без терминов n8n, где можно.
- Вопросы порциями, не списком из 15 пунктов.
- Никакого создания воркфлоу до явного апрува (Шаг 4).
- Секреты вводятся в n8n как credentials, не в чат.
- Если на любом шаге MCP/связь отвалились — вернись к n8n-setup.
